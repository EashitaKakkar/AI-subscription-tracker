### 1. What was the most uncomfortable trade-off you made because of the time pressure?
The most uncomfortable technical trade-off was omitting a secure administrative access control protocol (RBAC or API key protection) for the `/api/detect-changes` endpoint and public admin simulation panel. Leaving an endpoint exposed that alters pricing snapshots and triggers bulk database operations is a severe production vulnerability. However, within the 36-hour window, the choice came down to prioritizing a production-grade authentication layer or delivering a bulletproof side-by-side frontend diff view with complex data-drilling loop logic. I chose to protect the core visual and operational features required by the evaluation panel rather than getting bogged down in authentication middleware configuration.

### 2. If we extended the deadline by another 24 hours right now, what's the first thing you'd do?
I would refactor the initialization of the database client inside the change detection background script to use a secure server-side Service Role Key instead of the public anonymous key. Currently, querying user rows across multiple separate profiles using the anonymous client requires entirely opening up table permissions. With an extra 24 hours, I would implement robust Row-Level Security (RLS) configurations on Supabase and route this specific administrative task through a privileged background context. This would decouple our administrative automation workflow from frontend security constraints without relying on loose data permissions.

### 3. Looking back at your Round 1 codebase as a now-experienced user of it: what's one thing your Round 1 self made harder for your Round 2 self?
My Round 1 self designed a deeply nested hierarchical JSON dictionary for the application's pricing structure rather than a flat, sequentially indexed collection. Because prices were stored deeply inside multi-tiered billing period properties (`monthly` and `yearly` keys), my Round 2 self was forced to construct highly defensive conditional data-drilling logic. This structure made looping through historical rows and tracking down pricing changes incredibly error-prone, requiring extensive manual type-casting to safely extract numerical metrics and avoid running into fatal string-parsing blockers like `NaN` runtime values. Also the previous schema already stored the input_stack in the "advice" column, so I had to operate using the values present in the "advice" column instead of the input_stack, though it didn't affect the coding much, but sometimes I felt lost.
